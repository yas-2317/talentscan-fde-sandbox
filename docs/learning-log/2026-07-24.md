# 2026-07-24｜Git・GitHub・commit・push

## 今日の到達点

- ファイル保存とGitの履歴管理の違いを説明できた。
- `commit`と`push`が記録する場所の違いを理解した。
- `git status`と`git diff`で変更範囲を確認してからcommitできた。
- GitHubへのpushからVercel公開までの流れを実践した。

## ファイル保存とGitの違い

VS Codeで保存すると、Mac上にある現在のファイル内容が更新される。自動保存が有効な場合も、内部では同じファイル更新が行われる。

```text
VS Codeで編集
↓
保存または自動保存
↓
Mac上のファイル内容が更新
↓
Next.jsが変更を検知
↓
localhostへ反映
```

この時点では、Gitの履歴やGitHubはまだ更新されていない。Gitは保存前後の差分を認識し、選んだ変更を履歴として管理する。

## GitとGitHubの違い

| 名前 | 役割 |
|---|---|
| Git | 自分のPC内でファイルの変更履歴を管理する仕組み。 |
| GitHub | Gitで作った履歴をクラウド上に保存・共有するサービス。 |

Gitだけでもローカルの履歴は管理できる。GitHubへ反映するには、ローカルで作ったcommitをpushする必要がある。

## commitとpushの違い

| 操作 | 何をするか | 反映先 |
|---|---|---|
| `git add` | 次のcommitへ含める変更を選ぶ。 | staging area |
| `git commit` | 選んだ変更を履歴として確定する。 | ローカルGit |
| `git push` | ローカルのcommitを送る。 | GitHub |

```text
ファイル変更
↓
git add
↓
git commit
↓
ローカル履歴に保存
↓
git push
↓
GitHubへ反映
```

commitしただけではGitHubは更新されない。pushによって、初めてGitHub側へ履歴が届く。

## 今回実践した流れ

```text
VS Codeで編集・保存
↓
git status
↓
git diff
↓
git add
↓
git commit
↓
git push
↓
GitHub更新
↓
Vercelが自動デプロイ
↓
公開サイトへ反映
```

| コマンド | 今回確認・変更したこと |
|---|---|
| `git status` | 変更されたファイルとcommit対象の状態を確認した。 |
| `git diff` | 前回のcommitとの差分を確認した。 |
| `git add app/learning/layout.tsx` | 対象ファイルを次のcommitへ含める変更として選んだ。 |
| `git commit` | 変更をローカル履歴として確定した。 |
| `git push` | commitをGitHubへ送った。 |

`git status`は、作業前、commit前、push後に確認する。`git diff`では、意図した変更だけが含まれているかを見る。

## リポジトリで確認できた変更

対象ファイルは`app/learning/layout.tsx`である。実際のcommit `1ee17db`では、学習ポータルのナビゲーションを次のように変更していた。

| 変更前 | 変更後 |
|---|---|
| ダッシュボード | Dashboard |
| 教材 | Lessons |
| 学習ログ | Logs |
| トラブルログ | Troubleshooting |

commitメッセージは`changed tabs to english`で、GitHubの`main`へpush済みだった。Vercelのデプロイも成功し、公開サイトで英語のナビゲーションが表示されることを確認した。

公開サイト：<https://talentscan-fde-sandbox.vercel.app/learning>

手元の説明にあった「FDE LearningからFDE Learning Dashboardへの変更」は、現在のファイルとGit履歴では確認できなかった。実際の差分を記録するときは、記憶だけに頼らず`git diff`、`git log`、`git show`で確認する。

## ファイルパスとTab補完

長いファイルパスは、途中まで入力して`Tab`を押すと補完できる。

```bash
git add app/learning/lay
```

候補が1つなら、`app/learning/layout.tsx`まで補完される。候補が複数あるときに一覧が表示されるのはエラーではなく、追加の入力で候補を絞る必要がある状態である。

実務では、Tab補完を使いながら対象ファイルを明示すると、入力ミスと意図しないstageを減らせる。

## `git add .`との違い

```bash
git add .
```

このコマンドは、現在のディレクトリ以下にあるすべての変更をcommit対象へ追加する。便利だが、関係のない変更を含める可能性がある。

初学者のうちは、次の順番を優先する。

```text
git statusで対象確認
↓
git diffで差分確認
↓
対象ファイルを指定してgit add
↓
再度git statusで確認
```

## 基本コマンドを使う順番

| コマンド | 確認すること |
|---|---|
| `pwd` | 今いるディレクトリ |
| `ls` | 現在地にあるファイルとディレクトリ |
| `git status` | branchと変更状態 |
| `git diff` | 変更前後の差分 |
| `git add <file>` | 次のcommitへ含める変更 |
| `git commit -m "..."` | ローカル履歴の確定 |
| `git push` | GitHubへの送信 |
| `git log --oneline -5` | 直近5件のcommit |

コマンドの丸暗記より、「現在地を確認し、差分を見て、含める変更を選び、履歴を確定して共有する」という順番を理解する。

## 今日説明できるようになったこと

VS Codeの保存は現在のファイルを更新する操作であり、Gitはその変更を履歴として管理する。`git add`でcommit対象を選び、`git commit`でローカル履歴として確定し、`git push`でGitHubへ送る。今回の構成では、GitHubの`main`更新をきっかけにVercelが自動デプロイし、公開サイトへ反映した。
