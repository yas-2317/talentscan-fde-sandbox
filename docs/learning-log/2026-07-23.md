# 2026-07-23｜Next.js・localhostと障害切り分け

## 今日の到達点

- ReactとNext.jsの役割の違いを説明できた。
- `pnpm dev`から開発サーバーが起動する流れを理解した。
- `localhost`とポート番号が何を示すかを説明できた。
- Network、Console、サーバーログを組み合わせて障害箇所を絞り込めた。

## ReactとNext.jsの違い

| 技術 | 役割 |
|---|---|
| React | 画面をコンポーネント単位で作るためのライブラリ。 |
| Next.js | Reactを土台に、ページ管理、サーバー処理、API、ビルド、開発サーバーなどを扱うフレームワーク。 |

Next.jsはReactを単にまとめたpackageではなく、Reactを使ってWebアプリ全体を構築するための土台である。

## 開発サーバーとlocalhost

`package.json`には、次のscriptが定義されている。

```json
"dev": "next dev"
```

`pnpm dev`を実行すると、pnpmがこの設定を読み、Node.js上でNext.jsの開発サーバーを起動する。開発サーバーは、開発中のアプリを自分のPC内で動かすWebサーバーである。

| 要素 | 意味 |
|---|---|
| `localhost` | 自分自身のPCを指す特別な名前。 |
| `3000` | PC内のどのアプリへ通信を届けるかを区別するポート番号。 |
| `localhost:3000` | 自分のPCの3000番ポートで待つアプリへの接続先。 |

ポート番号が4桁であること自体に意味はない。Next.jsでは3000、Viteでは5173がよく使われるが、どちらも固定ではない。Viteは、Reactなどのフロントエンド開発で開発サーバー、コード変換、ビルドを担うツールである。TalentScanはNext.jsを使うため、通常は`localhost:3000`で確認する。

## ブラウザから画面が表示されるまで

```text
ブラウザ
↓ HTTPでアクセス
localhost:3000
↓
Next.jsの開発サーバー
↓
Node.js上で処理
↓
結果をブラウザへ返す
```

ブラウザがNext.jsを直接実行しているのではない。ブラウザは、Node.js上で動くNext.jsの開発サーバーへHTTPでアクセスし、処理結果を受け取って表示する。

## 処理の流れを理解する意味

TalentScanの処理は、次のような層に分けて考えられる。

```text
ブラウザ → localhost → Next.js → Node.js → API → DB
```

処理の流れを知る目的は用語の暗記ではなく、「どこまでは正常で、どこから先が怪しいか」を切り分けることである。

| 症状 | 最初に確認する場所 |
|---|---|
| `localhost:3000`が開かない | 開発サーバー、Node.js、ポート |
| 画面は出るが一部の操作だけ動かない | React側のコード、イベント処理、Console |
| API通信が500になる | API処理、DB接続、環境変数、サーバーログ |
| 一覧は出るが並び替えだけ失敗する | ブラウザ内のデータ型、ソート処理 |

FDEでは、顧客の「画面が動かない」という報告を、ブラウザ、API、DB、外部サービスなどの層へ分解して調べることが重要になる。

## ログを見る視点

ログは単なるエラー文ではなく、処理がどこまで到達したかを示す痕跡である。

- どの層のログか。
- どこまで成功した証拠か。
- 次に出るはずだったログは何か。

| 層 | 主な確認場所 |
|---|---|
| ブラウザ | DevToolsのConsole、Network |
| Next.js・Node.js | `pnpm dev`を実行しているターミナル、サーバーログ |
| API・DB | APIログ、DB接続エラー、外部サービスのレスポンス |
| 本番環境 | VercelのBuild Logs、Runtime Logs |

## Networkとサーバーログの違い

Networkはブラウザから発生した通信の痕跡、サーバーログはサーバー側で処理が実行された痕跡である。Networkに新しい通信がなければ、その操作ではブラウザから新しいHTTP通信を送っていないと判断できる。

ただし、定期処理、cron、queue、webhook、起動時の初期化、再コンパイル、バックグラウンド処理では、ブラウザの通信がなくてもサーバーログが出る。

```text
NetworkにAPI通信なし
サーバーログなし
Consoleにエラーあり
→ ブラウザ側で止まっている可能性が高い
```

```text
NetworkにAPI通信あり
Status 500
サーバーログにDB接続エラー
→ APIまでは到達し、DB接続付近で失敗
```

## 切り分け練習

### 練習1：候補者一覧APIが500

`GET /api/candidates`がStatus 500で、サーバーログに`database connection failed`が出た。ブラウザからAPIを呼ぶところまでは到達しており、APIからDBへの接続で失敗している可能性が高い。

次に、DBの起動状態、接続先、環境変数、接続URL、ユーザー名、パスワード、API内のDB呼び出し処理、詳細なサーバーログを確認する。

### 練習2：高評価順に並ばない

一覧の初期表示は成功しているが、ボタンを押してもNetworkとサーバーログに変化がなく、Consoleに`TypeError: candidates.sort is not a function`が出た。APIやDBより先に、ブラウザ側の並び替え処理を確認する。

```javascript
console.log(candidates)
console.log(Array.isArray(candidates))
```

`sort`は配列に使うため、`candidates`が配列かを確認する。オブジェクトであれば、内部の配列を正しく指定する必要がある。

## 今日説明できるようになったこと

一連の処理の流れを理解すると、問題発生時に、どこまでは正常でどこから先が怪しいかを切り分けられる。Network、Console、サーバーログを組み合わせることで、ブラウザ側、API、DB、外部サービスのどの層で止まっているかを絞り込める。
